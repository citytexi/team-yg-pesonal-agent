# LLM Wiki Schema

이 문서는 LLM(Claude Code)이 위키를 운영하는 방식을 정의한 스키마다.
대화 시작 시 이 파일을 먼저 읽고 모든 규칙을 따를 것.

## 디렉토리 구조

```
raw/              ← 불변 원본. 절대 수정 금지. 읽기만 가능.
  assets/         ← 로컬 이미지 및 첨부파일
wiki/             ← LLM이 생성·관리하는 전체 위키
  index.md        ← 전체 페이지 카탈로그 (항상 최신 상태 유지)
  log.md          ← append-only 이력 로그
  overview.md     ← 위키 전체를 관통하는 살아있는 개요/논지 (핵심)
  entities/       ← 사람, 조직, 제품, 프로젝트 등 고유 엔티티
  concepts/       ← 개념, 용어, 기술, 이론 등 주제 페이지
  sources/        ← raw/ 원본별 요약 페이지
  synthesis/      ← 쿼리 결과, 분석, 비교표, lint 보고서
    open-questions.md  ← 미해결 모순·오픈 질문 추적
```

---

## 공통 규칙

1. **raw/ 불변**: `raw/`는 읽기 전용. 내용을 수정하거나 파일을 생성/삭제하지 않는다.

2. **Obsidian 링크**: 내부 링크는 **항상 `[[파일명]]` 형식만 사용**한다.
   - 경로(`[[wiki/entities/파일명]]`) 또는 서브폴더(`[[entities/파일명]]`) 접두사를 붙이지 않는다.
   - 파일명이 충돌할 때만 예외적으로 경로를 추가한다.
   - 파일명 규칙: 영문 kebab-case 기본. 단, **personal 도메인의 한국 인물·장소 등 고유명사는 한글 파일명 허용** (`[[이효리]]`, `[[서울]]`).

3. **YAML frontmatter**: 모든 wiki/ 파일에 아래 형식을 포함한다.
   ```yaml
   ---
   tags: [카테고리, 도메인태그]
   sources: [원본파일명.md]
   updated: YYYY-MM-DD
   ---
   ```

4. **작업 완료 시**: 매번 `wiki/index.md`와 `wiki/log.md`를 업데이트한다.

5. **언어**: 사용자와 한국어로 대화하되, 위키 내용은 한국어로 작성.

6. **개인 데이터 정책 (⚠️ 이 저장소는 public remote)**:
   - 이 repo는 공개(public)다. 추적되는 모든 파일은 인터넷에 평문으로 노출된다.
   - **식별 가능한 개인정보는 절대 추적되는 `wiki/`에 커밋하지 않는다.** 실명·연락처(이메일/전화)·주소/위치·생년월일·건강·재무·계정/인증 정보 등.
   - 이런 콘텐츠는 반드시 `wiki/personal-private/`(`.gitignore`로 추적 제외)에 저장한다.
   - 추적되는 `wiki/`에는 공개돼도 무방한 일반 지식·비식별 콘텐츠만 둔다.
   - `personal` 태그 페이지를 만들 때는 기본적으로 `wiki/personal-private/`에 생성하고, 공개 가능함이 명확한 경우에만 추적 영역으로 옮긴다.
   - 커밋 전 항상 lint를 돌려 민감정보 노출을 점검한다. 발견 시 커밋을 중단하고 사용자에게 보고한다.

## 페이지 생성 임계 기준 (granularity)

독립 페이지를 **만드는** 조건 (둘 중 하나 충족):
- 2개 이상의 소스에서 등장한다
- 단독으로 다시 참조할 가능성이 높다 (핵심 개념, 핵심 인물 등)

독립 페이지를 **만들지 않는** 조건:
- 한 소스에서만 1회 언급된 부수적 요소 → 관련 페이지 내 인라인 언급 + 백링크로 처리
- stub 수준(이름만 있고 내용이 없을 것이 확실한 경우)

## 워크플로우 1: ingest (인터랙티브)

트리거: 사용자가 `raw/` 경로의 파일 또는 텍스트를 하나씩 주입할 때.

### 단계

1. **읽기**: 대상 파일을 읽는다. 이미지가 포함되어 있으면 텍스트를 먼저 처리하고, 필요한 이미지를 이후 단계에서 별도로 본다 (→ [이미지 처리 규칙](#이미지-처리-규칙) 참고).
2. **논의**: 핵심 내용, 발견한 엔티티·개념을 사용자와 간략히 논의한다.
3. **sources/ 페이지 생성**: `wiki/sources/[파일명].md`에 요약 페이지를 작성한다.
4. **entities/ 업데이트**: 페이지 생성 임계 기준을 만족하는 엔티티만 페이지 생성 또는 병합.
   - 모순 발견 시: 해당 섹션에 `> ⚠️ [날짜] [[원본파일]] 과 상충` 마커 추가 **+ `open-questions.md`에 항목 등록**.
5. **concepts/ 업데이트**: 임계 기준을 만족하는 개념만 신규 생성 또는 업데이트.
6. **overview.md 갱신**: 이 소스가 현재 논지를 강화하거나 반박하면 `wiki/overview.md`를 업데이트한다.
7. **index.md 업데이트**: 새 페이지를 카탈로그에 추가.
8. **log.md 추가**: `## [YYYY-MM-DD] ingest | [원본 제목]` 형식으로 한 줄 기록.

### 워크플로우 1b: ingest (배치)

트리거: 사용자가 여러 소스를 한꺼번에 주입할 때 ("배치로", "한꺼번에" 등의 표현).

- 단계 2(논의)를 **생략**하고 나머지를 반복 실행한다.
- 전체 완료 후 요약 보고서를 사용자에게 한 번에 제공한다: "N개 소스 처리, M개 페이지 생성/업데이트, K건 모순 발견".
- 모순이나 불명확한 판단이 필요한 경우만 중간에 사용자에게 확인한다.

### sources/ 페이지 템플릿

```markdown
---
tags: [source, 도메인]
sources: [원본파일명.md]
updated: YYYY-MM-DD
---

# [원본 제목]

- **원본**: `raw/파일명.md`
- **날짜**: YYYY-MM-DD
- **도메인**: research | personal

## 핵심 요점
- ...

## 관련 엔티티
- [[엔티티명]]

## 관련 개념
- [[개념명]]
```

## 워크플로우 2: query

트리거: 사용자가 위키 전체를 바탕으로 질문하거나 분석을 요청할 때.

### 단계

1. **index.md 읽기**: 먼저 `wiki/index.md`를 읽어 관련 페이지를 파악한다.
2. **관련 페이지 읽기**: index에서 파악한 관련 페이지들을 읽는다.
3. **출력 형식 선택**: 질문 성격에 따라 아래 형식 중 하나를 선택한다.
   - 일반 분석 → 마크다운 본문
   - 항목 비교 → 비교표 (마크다운 테이블)
   - 발표용 → Marp 슬라이드 (frontmatter에 `marp: true`)
   - 데이터 시각화 → matplotlib 코드 블록
4. **답변 작성**: 읽은 내용을 종합해 출처 링크를 포함한 답변을 제공한다.
5. **저장 여부 판단**: 재사용 가치가 있는 분석·비교 결과는 `wiki/synthesis/[주제].md`에 저장하고 `overview.md` 갱신 필요 여부를 판단한다.
6. **log.md 추가**: `## [YYYY-MM-DD] query | [질문 요약]` 형식으로 기록.

## 워크플로우 3: lint

트리거: 사용자가 "위키 상태 점검" 또는 "lint"를 요청할 때.

### 점검 항목

1. **민감 데이터**: 이메일, 전화번호, API 키, 비밀번호 패턴 탐지.
   - **발견 시**: 자동 수정 금지. 사용자에게 파일명·줄 위치를 보고하고 처리 방법(삭제·마스킹·private 폴더 이동)을 확인한다.
2. **모순 감지**: 같은 주제를 다루는 페이지들 간 상충 내용 탐지 → `open-questions.md`에 미해결 항목 등록.
3. **고아 페이지**: 어떤 페이지에서도 `[[링크]]`되지 않은 wiki/ 파일.
4. **링크 없는 언급**: 페이지 없이 언급만 되는 엔티티·개념 (→ 페이지 생성 권고 또는 임계 기준 미달이면 무시).
5. **데이터 공백**: 채울 수 있는 빈 영역 또는 조사 권고 주제.
6. **stale 정보**: 최근 원본과 모순되는 오래된 내용.

### 출력

`wiki/synthesis/lint-YYYY-MM-DD.md`에 보고서 저장.

```markdown
---
tags: [lint, maintenance]
updated: YYYY-MM-DD
---

# Lint 보고서 YYYY-MM-DD

## 민감 데이터 ⚠️
- (없음 또는 파일명:줄번호 목록 — 처리 전까지 커밋 금지)

## 모순 → open-questions.md 등록됨
- [[페이지A]] vs [[페이지B]]: ...

## 고아 페이지
- [[고아파일명]]

## 링크 없는 언급
- "개념명" → 페이지 생성 권고 (임계 기준 충족 시)

## 데이터 공백 / 조사 권고
- ...
```

## 모순 해소 워크플로우 (open-questions.md)

`wiki/synthesis/open-questions.md`는 모순·미결 질문의 단일 추적 장소다.

항목 형식:
```markdown
### [YYYY-MM-DD] [주제 요약]
- **출처 A**: [[페이지A]] — 주장 요약
- **출처 B**: [[페이지B]] — 상충 주장 요약
- **상태**: 미해결 | 해소됨 | 보류
- **해소 메모**: (해소된 경우 어느 쪽이 옳고 왜인지)
```

ingest·lint에서 모순이 발견되면 이 파일에 항목을 추가한다.
해소되면 상태를 "해소됨"으로 바꾸고 관련 페이지에서 `⚠️` 마커를 제거한다.

## 이미지 처리 규칙

- raw/assets/ 에 저장된 이미지는 ingest 시 **텍스트 처리 후** 필요한 이미지만 별도로 본다.
- 단순 장식·배너 이미지는 건너뛴다.
- 이미지에 데이터·도표·텍스트가 포함된 경우 해당 이미지를 별도로 읽고 핵심 내용을 sources 페이지에 기록한다.
- **텍스트 전용 소스만 다루는 경우**: assets/ 폴더를 사용하지 않고, ingest에서 이미지 단계를 생략한다.

## 확장성 경로

현재는 `wiki/index.md` 파일 하나로 탐색을 지원한다.
- **~100개 소스 이하**: index.md 방식으로 충분.
- **100개 초과 시**: [qmd](https://github.com/tobi/qmd) (로컬 BM25+벡터 하이브리드 검색) 도입을 고려한다. CLI 또는 MCP 서버로 연결해 LLM이 직접 쿼리할 수 있다.

## 도메인 태그 규칙

| 태그 | 설명 | 예시 |
|------|------|------|
| `research` | 기술·학문적 주제 | LLM, 자율주행, 논문, 기술 아티클 |
| `personal` | 개인 성장·기록 | 목표, 독서, 회고, 인사이트 |

하나의 페이지가 두 도메인에 걸칠 수 있다.
이 repo는 public이므로 식별 가능한 personal 콘텐츠는 기본적으로 `wiki/personal-private/`에 저장한다 (git 추적 제외). 공통 규칙 6 참고.

## index.md 구조

```markdown
# Wiki Index

> 마지막 업데이트: YYYY-MM-DD | 총 페이지 수: N

## Overview
- [[overview]] — 위키 전체 논지 요약

## Sources (원본 요약)
- [[파일명]] — 한줄 요약

## Entities (엔티티)
- [[이름]] — 한줄 설명

## Concepts (개념)
- [[개념명]] — 한줄 설명

## Synthesis (분석·결과)
- [[파일명]] — 한줄 설명
```

## log.md 구조

append-only. 새 항목은 파일 **끝에** 추가.

```
## [YYYY-MM-DD] ingest | 제목
## [YYYY-MM-DD] query | 질문 요약
## [YYYY-MM-DD] lint | 점검 완료, N건 발견
```

`grep "^## \[" wiki/log.md | tail -10` 으로 최근 10개 이력 확인 가능.
